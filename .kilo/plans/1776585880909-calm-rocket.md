# Plan: Per-Project Stats Fetching and User-Specific Metrics

## Summary

Extend the `/metrics` endpoint to fetch per-project stats from the ArchiveTeam tracker API, extract both overall and user-specific metrics, cache the results, and expose them as Prometheus gauges.

## Context

The stats endpoint (`https://v1.api.tracker.archiveteam.org/<name>/stats.json`) returns:

```json
{
  "downloaders": ["user1", "user2"],
  "downloader_bytes": {"user1": 12345, "user2": 67890},
  "downloader_count": {"user1": 100, "user2": 200},
  "domain_bytes": {"domain": 12345678},
  "total_items_todo": 160018,
  "total_items_out": 1049118,
  "total_items": 20669655,
  "counts": { "done": 19460519, "rcr": 0.45, "out": 1049118, "todo": 160018 },
  "total_items_done": 19460519
}
```

## Steps

### 1. Create `ProjectStatsResponse` model

**File:** `src/ArchiveTeam.Exporter.ApiService/Models/ProjectStatsResponse.cs`

New model class to deserialize the stats.json response:

```
ProjectStatsResponse
├── Downloaders: string[]
├── DownloaderBytes: Dictionary<string, long>
├── DownloaderCount: Dictionary<string, long>
├── DomainBytes: Dictionary<string, long>
├── TotalItemsTodo: long
├── TotalItemsOut: long
├── TotalItems: long
├── Counts: ProjectStatsCounts
│   ├── Done: long
│   ├── Rcr: double
│   ├── Out: long
│   └── Todo: long
└── TotalItemsDone: long
```

Parse all fields from the API response into the model (including `downloaders`, `domain_bytes`, `counts.rcr`) for future use, even though not all will be exposed as Prometheus metrics initially. Use `JsonNamingPolicy.SnakeCaseLower` during deserialization.

### 2. Update `IProjectService` interface

**File:** `src/ArchiveTeam.Exporter.ApiService/Services/IProjectService.cs`

Add a new method:

```csharp
Task<ProjectStatsResponse?> FetchProjectStatsAsync(string projectName, CancellationToken cancellationToken);
```

### 3. Update `ProjectService` implementation

**File:** `src/ArchiveTeam.Exporter.ApiService/Services/ProjectService.cs`

#### 3a. Add Prometheus gauge definitions

Define gauges at the top of the class (next to existing `_projectInfoGauge`):

- `archiveteam_project_total_items` — label: `name` (overall item count for the project)
- `archiveteam_project_items_done` — label: `name` (completed items)
- `archiveteam_project_items_todo` — label: `name` (remaining items)
- `archiveteam_project_items_out` — label: `name` (items currently being worked on)
- `archiveteam_project_user_bytes` — labels: `name`, `username` (bytes downloaded by the user for this project)
- `archiveteam_project_user_items` — labels: `name`, `username` (items downloaded by the user for this project)

#### 3b. Implement `FetchProjectStatsAsync`

- Cache key pattern: `stats_{projectName}`
- Cache duration: same as `_options.CacheDuration`
- Use the same cache-with-lock pattern as `FetchProjectsAsync` (single `SemaphoreSlim` or a `ConcurrentDictionary<string, SemaphoreSlim>` — for simplicity, use the existing `_cacheLock` since requests are sequential)
- HTTP GET to `https://v1.api.tracker.archiveteam.org/{projectName}/stats.json`
- Deserialize using `JsonNamingPolicy.SnakeCaseLower`
- Return the deserialized response, cached

#### 3c. Update `GetProjectGaugesAsync`

After getting the filtered projects list and setting the existing info gauge, iterate over each project:

1. Call `FetchProjectStatsAsync(project.Name, cancellationToken)`
2. If stats are returned:
   - Set overall gauges: `total_items`, `items_done`, `items_todo`, `items_out`
   - Look up the configured `_options.Username` in `DownloaderBytes` and `DownloaderCount`
   - If found, set user-specific gauges: `user_bytes`, `user_items`

### 4. No changes needed to `Program.cs`

The `/metrics` endpoint already calls `GetProjectGaugesAsync` and exports the registry. No structural changes required.

### 5. No changes needed to `ArchiveTeamOptions`

The `Username`, `CacheDuration`, and `ProjectsWhitelist` are already configured.
